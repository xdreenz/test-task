# Вопрос 1 : В BaseTest используется implicitlyWait(10s). Ваш коллега добавил явный WebDriverWait(driver, 15s) в одном из тестов. При запуске тесты стали нестабильными - иногда падают, иногда нет, без видимой причины. Объясните возможную механику происходящего и как бы вы исправили проблему.

Документация Selenium прямо запрещает смешивать implicitlyWait и WebDriverWait:

> *Warning: Do not mix implicit and explicit waits. Doing so can cause unpredictable wait times.*

Разберу почему.

## Что делает каждый тип wait

### Implicit wait (10s)

Настройка драйвера: каждый вызов `findElement` / `findElements`, который не нашёл элемент сразу, будет внутри себя пуллить DOM до 10 секунд, прежде чем кинуть `NoSuchElementException`.

Применяется на уровне команды поиска, прозрачно для теста.

### Explicit wait (`WebDriverWait`, 15s)

Цикл вида “вызывай `ExpectedCondition` каждые ~500ms, пока не вернёт truthy или не истечёт таймаут”. Внутри условий типа `visibilityOfElementLocated` или `presenceOfElementLocated` сидит тот же `findElement`.

* * *

## Где они будут конфликтовать

### Сценарий 1: ожидание появления элемента - суммируются таймауты

```java
new WebDriverWait(driver, Duration.ofSeconds(15))
    .until(ExpectedConditions.presenceOfElementLocated(By.id("foo")));
```

Внутри presenceOfElementLocated вызывается `driver.findElement(By.id("foo"))`. Этот вызов уже сам ждёт 10 секунд из-за implicit wait.

Что получается:

- Элемент не появился -> findElement висит 10 секунд -> бросает исключение -> WebDriverWait ловит, делает **одну** итерацию, ждёт ~500ms -> следующая итерация -> ещё 10 секунд findElement ->…
- Реальный таймаут до падения: не 15 секунд, а ближе к 20+ (две итерации по ~10 секунд). Иногда успеет три попытки, иногда две, отсюда плавающее поведение.
- Если элемент в итоге появляется на 12-й секунде, тест может либо успеть его поймать, либо нет - зависит от того, в какой момент цикла WebDriverWait это произошло.

**Откуда нестабильность:** время между retry внутри WebDriverWait теперь не ~500ms, а ~10 секунд. Тайминги становятся грубыми, и любой network-jitter решает, проходит тест или нет.

### Сценарий 2: ожидание **отсутствия/невидимости** элемента

```java
wait.until(ExpectedConditions.invisibilityOfElementLocated(By.id("bar")));
```

Здесь происходит инверсия ожидаемого поведения:

- Элемент уже исчез, findElement его не находит, но implicit wait заставляет драйвер ждать 10 секунд, надеясь что элемент появится, только потом возвращается с NoSuchElementException, invisibilityOfElementLocated трактует это как "успех, элемента нет".
- Тест, который должен пройти за 50мс, проходится за 10 секунд.

А в случае `numberOfElementsToBe(locator, 0)` или кастомных условий через findElements (который не бросает исключение, а возвращает пустой список) - implicit wait всё равно ждёт 10 секунд на каждой проверке.

### Сценарий 3: stale element

WebDriverWait обычно ловит StaleElementReferenceException и ретраит. Но при включённом implicit wait повторный findElement снова может ждать 10 секунд до момента, пока элемент не реинициализируется в DOM. На быстрых страницах работает, на медленных (или под нагрузкой CI) - нет.

* * *

## Как чинить

### Шаг 1: исключить implicitlyWait

Implicit wait - глобальное состояние драйвера, которое нельзя локально отключить в одном тесте без побочных проблем.

### Шаг 2: все ожидания - только explicit, через Page Object

```java
public class AdminUsersPage {
    private static final Duration DEFAULT_TIMEOUT = Duration.ofSeconds(10);
    private final WebDriverWait wait;

    public AdminUsersPage(WebDriver driver) {
        this.wait = new WebDriverWait(driver, DEFAULT_TIMEOUT);
    }

    public WebElement userRow(String id) {
        return wait.until(ExpectedConditions
            .visibilityOfElementLocated(By.cssSelector("tr[data-user-id='" + id + "']")));
    }

    public boolean isUserAbsent(String id) {
        return wait.until(ExpectedConditions
            .invisibilityOfElementLocated(By.cssSelector("tr[data-user-id='" + id + "']")));
    }
}
```

### Шаг 3: единая точка конфигурации таймаутов

```java
public final class Waits {
    public static final Duration SHORT  = Duration.ofSeconds(2);
    public static final Duration NORMAL = Duration.ofSeconds(10);
    public static final Duration LONG   = Duration.ofSeconds(30);

    public static WebDriverWait normal(WebDriver d) {
        return new WebDriverWait(d, NORMAL);
    }
    private Waits() {}
}
```

# Вопрос 2 : Тест-сьют из 200 UI тестов стабильно проходит локально, но в CI падает ~15% тестов с разными ошибками: StaleElementReferenceException, NoSuchElementException, и иногда - неожиданный 401 от API. Тесты перезапускаются и проходят. Какие 3 наиболее вероятные причины нестабильности вы бы проверили в первую очередь, и как бы вы их диагностировали?

Это классическое состояние гонки на медленном/нагруженном окружении, а не функциональный баг. Поэтому я бы копал не в логику тестов, а в три нижеперечисленных направления, в порядке убывания вероятности.

## Причина №1: Параллелизм тестов делит общее состояние

- Локально часто гоняется один тест или малая группа - нет конкуренции.
- Если конфиг для скорости, то тесты начинают конкурировать за ресурсы, которые никто не считал разделяемыми.
- 401 от API в UI-тестах почти всегда означает, что один тест разлогинил/удалил пользователя, которого использует другой.
- StaleElement и NoSuchElement тоже укладываются в эту картину: один тест триггерит редирект/перерисовку, второй в это время работает с DOM.

### Как диагностировать

1.  **Убрать параллелизм.**
    
    Если флаки исчезли - причина найдена.
    
2.  **Запустить с фиксированным seed-порядком и сравнить:**
    
    ```
    -Djunit.jupiter.testmethod.order.default=org.junit.jupiter.api.MethodOrderer$Random
    -Djunit.jupiter.execution.parallel.enabled=false
    ```
    
    и тот же seed несколько раз. Если падают **одни и те же пары** в одном порядке - это межтестовая зависимость.
    
3.  **Проверить код на маркеры shared state:**
    
    - статические поля в test-utils,
    - `RestAssured.requestSpecification = ...`,
    - singleton’ы (`getInstance`),
    - использование одинаковых хардкод-логинов/email’ов.
4.  **Логировать `Thread.currentThread().getName()` + test name + userId** - в CI-логах сразу видно, два ли теста одновременно ходят за одного юзера.
    
5.  **Включить изоляцию каждого теста:**
    
    - уникальные email через UUID,
    - свой токен/сессия на тест,
    - cleanup в @AfterEach.

### Как чинить

- Каждому тесту - генерировать на лету свои тестовые данные.
- Никаких static mutable полей.
- Если параллелизм нужен - @Execution(CONCURRENT) с честной test data factory.

## Причина №2: Таймауты, рассчитанные под локальную машину, неприменимы к CI

- Локально CPU свободен, задержка до бэкенда - <span style="color: rgb(0, 0, 0);"><span style="color: #ffffff;">околонулевая</span></span>.
- В CI: shared runner, контейнеры, сетевые хопы между сервисами, headless <span style="color: rgb(0, 0, 0);">Chrome</span> конкурирует с тестами за CPU. Рендер страницы, который локально занимает 200ms, в CI может занять 3-5 секунд.
- StaleElement и NoSuchElement как раз и появляются, когда тест успел сделать действие до того, как страница дорендерилась/переинициализировала DOM.

### Как диагностировать

1.  **Сравнить время выполнения тестов.** Если в CI оно в разы больше - это системная разница в окружении.
    
2.  **Включить Selenium tracing** и посмотреть, сколько реально занимает findElement, click, navigate.
    
3.  **На упавшем тесте сделать screenshot + page source + browser console log** в @AfterEach
    
    Часто на скриншоте можно увидеть спиннер, это сразу подтверждает гипотезу.
    

### Как чинить

- Исключить `Thread.sleep`.
    
- Все ожидания - WebDriverWait с таймаутом, рассчитанным на CI (×2-3 от локального).
    
- Для StaleElement - паттерн “найди заново внутри `wait.until`”, а не “найди один раз и кликни”.
    
- Для 401 - обновлять токен в каждом тесте (или хотя бы проверять перед использованием не протух ли он).
    
- Все таймауты брать из конфига, разные для local/ci профилей.
    

## Причина №3: Нестабильность стенда

- 401 “иногда” - это характерный симптом проблем с auth-сервисом, а не тестов. Если auth-сервис стартует медленнее, чем тесты начинают ходить - первые N запросов получают 401.
- Сеть между контейнерами в CI неидеальна.

### Как диагностировать

1.  **Делать Healthcheck перед прогоном тестов.**
    
2.  **Собрать логи всех контейнеров за время прогона** (`docker logs`, `kubectl logs --previous`) и грепнуть по OOMKilled, restart, panic, connection refused.
    
3.  **Корреляция падений по времени:** если 401 ловится только в первые 30 секунд прогона, значит auth-сервис не успел подняться. Если в середине - возможно, рестарт. Если рандомно у любого теста - token expiration / race.
    
4.  **Метрики runner’а во время прогона:** CPU steal time, memory pressure, disk I/O wait.
    
5.  **Запустить тот же CI-job 10 раз подряд без изменений кода** - если процент падений плавает (5%, 30%, 0%), это инфра, а не тесты. Если стабилен - это код.
    

### Как чинить

- Healthcheck + retry на старте всего сьюта, не retry внутри тестов как замена решения.
- Ресурсные лимиты на контейнеры в CI (фиксированный CPU/RAM, чтобы прогон был воспроизводим).
- try-with-resources или гарантированный `quit()` в @AfterEach (`if (driver != null)`) + cleanup-скрипт перед каждым джобом.
- Если используется Grid - мониторинг session timeout, перезапуск нод между прогонами.

# Вопрос 3 : Ниже приведен полный код теста (импорты опущены для краткости). Найдите в нем проблему и объясните её:

```public
@Test  
void testDeleteUser() {  
    String userId = “user-123”;  
    apiClient.deleteUser(userId);
    Response response = apiClient.getUserById(userId);
    assertThat(response.statusCode()).isEqualTo(404);
}
```

Тут сразу несколько проблем.

## Проблема №1. В тесте нет фазы Arrange

Такой тест нельзя считать проверкой удаления, если неизвестно есть ли что удалять. Возможные причины ложных прохождений теста:

- Если пользователь user-123 никогда не существовал, то DELETE также вернёт 404. Тест пройдёт успешно, хотя ничего не было удалено.
- Пользователя user-123 удалил предыдущий тест.
- В тестовой БД его вообще не должно быть.

## Проблема №2. Хардкод `userId = "user-123"`

Это ломает тест в нескольких аспектах:

### 1\. Тест не идемпотентный

- Первый прогон (если пользователь был): удалили, всё ок.
- Второй прогон: пользователя уже нет -> DELETE 404, GET 404 -> тест проходит по неправильной причине.
- На чистой БД: первый прогон сразу падает или проходит ложно.

### 2\. Тест не работает параллельно

Два инстанса TestDeleteUser дерутся за один и тот же user-123. Один удалит первым, второй получит 404 от DELETE, потом 404 от GET -> ложно зелёный, либо упадёт на 5xx, если API не идемпотентен.

### 3\. Тест зависит от внешнего состояния

Если user-123 приходит из миграции/seed-скрипта, значит у нас межтестовая связь через данные. Любой другой тест, удаляющий или меняющий этого юзера, ломает этот. Это классический источник “локально работает, в CI 15% падает”.

### 4\. Использование строкового хардкода вместо UUID

Имя user-123 намекает на антипаттерн - тесты неявно зависят от состояния БД. Будет правильнее генерировать данные на лету в фазе Arrange, и userId получать из ответа createUser.

## Проблема №3. Нет cleanup, нет teardown

Даже если тест поправить и начать создавать юзера в Arrange, то:

- Если `apiClient.createUser` упадёт - тест упадёт, юзер не создан, всё ок.
- Если упадёт `apiClient.deleteUser` (середина теста) - юзер останется в БД. Следующие прогоны будут засорять окружение.
- Если упадёт ассерт после delete - юзера, скорее всего, уже нет, но в граничном случае (DELETE вернул 200, а юзер на самом деле жив) останется.

Нужен @AfterEach или try/finally с deleteUserIfExists, толерантным к 404.
